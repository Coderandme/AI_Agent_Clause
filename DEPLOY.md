# Deploying

Everything on **Render**, both services described by one committed file, [`render.yaml`](render.yaml):

| Piece | Render service | Why that type |
|---|---|---|
| **API** (FastAPI, agent, ingest, retention) | **Web Service** (Docker, free) | needs a process that stays alive after it responds |
| **SPA** (React, and the bundled demo) | **Static Site** (free) | just files on a CDN; works with the API asleep |
| **Postgres** | **Neon** (free) | already set up |

**Total infrastructure cost: $0.** OpenAI tokens are the only bill, and the demo never touches them.

> **Why two service types rather than one box?** They have opposite requirements. An analysis takes
> 40–80 seconds and runs as a background task *after* the HTTP response, so the API needs a real,
> long-lived process — which is why it can't be serverless (Vercel, Lambda, Cloud Run's default all
> kill a function the moment it responds). The SPA is the opposite: static files that should sit on a
> CDN and never think. Same platform, two shapes.

> **Why not Hugging Face any more?** It was the plan until **July 2026**, when HF removed the free CPU
> Basic flavor and paywalled the Docker SDK for unpaid accounts. That is the second host to withdraw
> its free tier under this project (Fly.io was the first), which is why `render.yaml` exists: the
> deployment is a file, so moving hosts is an afternoon and not an archaeology project.

## 1. Create both services from the Blueprint

Render reads [`render.yaml`](render.yaml), so there is almost nothing to click.

1. **Push the repo to GitHub** (Render deploys from a branch, so it can only deploy what is pushed).
2. Render dashboard → **New** → **Blueprint** → pick the `AI_Agent_Clause` repo. It finds
   `render.yaml` and proposes **two** services: `clause-api` and `clause-web`, both free.
3. Render **prompts for the values marked `sync: false`**. Paste what you can; two of them are
   chicken-and-egg (each service needs the other's URL), so leave those blank and fill them in step 2:

   | Key | Service | Value |
   |---|---|---|
   | `DATABASE_URL` | api | your Neon connection string (the one in your local `.env`) |
   | `OPENAI_API_KEY` | api | your OpenAI key |
   | `CORS_EXTRA_ORIGINS` | api | *leave blank for now* |
   | `VITE_API_URL` | web | *leave blank for now* |

   `JWT_SECRET` and `IP_HASH_SECRET` are **generated by Render** and never typed by anyone. That is
   the correct handling: a secret nobody has seen cannot be leaked by a person.

4. Apply. The first API build takes a few minutes (it builds the image and installs from `uv.lock`).
   Migrations run automatically at boot, so the schema arrives with the code.
5. Check the API: `https://clause-api-xxxx.onrender.com/api/health` → `{"status":"ok","db":true,...}`

**The free API sleeps** after ~15 minutes idle; the next request waits ~50s. That is deliberate and it
never touches the demo — see the note in `render.yaml`.

## 2. Introduce the two services to each other

You now have both URLs, so fill in the two blanks. **Each needs a redeploy of its own service.**

| Set on | Key | Value |
|---|---|---|
| `clause-web` | `VITE_API_URL` | `https://clause-api-xxxx.onrender.com` |
| `clause-api` | `CORS_EXTRA_ORIGINS` | `["https://clause-web-xxxx.onrender.com"]` |

Two things that bite here:

- **`CORS_EXTRA_ORIGINS` is JSON** — brackets and quotes, because it is parsed as a list, not a
  string. Get it wrong and the API refuses to boot (loudly, which is the point).
- **`VITE_API_URL` is baked in at BUILD time**, not read at runtime. Setting it requires a *redeploy*
  of `clause-web`, not a restart.

Being on one platform does **not** remove the need for CORS: `clause-web` and `clause-api` are still
different origins, and the API locks CORS to an exact list — never `*`, because it carries the token
that gates spending.

Skip this step and the symptom is confusing but the cause is simple: the demo works perfectly (it
needs no backend at all) while **login fails with a CORS error**.

## 4. Invite yourself

Nobody can spend anything until a code exists. From your machine, against the same Neon database:

```bash
cd api
uv run python -m clause.auth.codes new --grant 3        # → CLAUSE-XXXX-XXXX
# sign up on the site with that code, then:
uv run python -m clause.auth.codes make-admin you@email.com   # unlimited, takes effect on refresh
```

## What it costs, and what stops it costing more

**OpenAI is the only bill**, and access is **invite-only** (SPEC.md §2.5):

| | limit |
|---|---|
| anonymous | cannot upload at all. Demo only, $0. |
| invited account | `upload_grant` analyses, lifetime |
| admin | unlimited |
| **hard ceiling** | **$5/month** — nothing crosses it, admin or not |

The ceiling is **armed**: every completed analysis records its cost to `usage_ledger`
(`analysis/service.py`), and `assert_can_upload` refuses once the month's total reaches
`monthly_ceiling_microdollars`. A scan costs ~$0.42 on `sol`, so $5 is about a dozen analyses.

Raise it in `api/clause/config.py`, or by setting `MONTHLY_CEILING_MICRODOLLARS` in Render.

**Uploads are deleted after 24 hours** by the retention sweep (`clause/retention.py`), which runs at
boot and hourly inside the API process. Because the free plan sleeps, the boot sweep is the important
one: it cleans up the moment the service wakes.

## Re-baking the demo

The pre-computed analyses are the output of a specific prompt, rule library, and model. Change any of
those and the demo shows a system that no longer exists:

```bash
cd api && uv run python -m clause.demo
```

Costs about $0.80 (two contracts on `sol`), then the demo is free forever again. Commit the updated
`demo/precomputed/*.json` — the SPA bundles them at build time, so `clause-web` needs a redeploy to
serve the new ones.
